# Schema Meta-Parser

`lib/pytorch_schema/pytorch_schema.ml` — parses `schema.yaml` into an OCaml type map.

## Why jsont for YAML?

`yamlt` (vendored) is a YAML-to-jsont bridge: it parses YAML and produces a
`Jsont.json` value. This lets the schema meta-parser use the same `jsont` API that
the generated code uses for `model.json`. One decoder API, two formats.

`Schema.of_yaml_string` calls `Yamlt.of_string jsont s` where `jsont : Schema.t Jsont.t`.

## Type Hierarchy

```
Schema.t
├── types : Type_def.t String_map.t   (the main payload)
├── schema_version : int * int
└── treespec_version : int

Type_def.t = Struct of Struct.t | Union of Union.t | Enum of Enum.t

Struct.t = { fields : Field.t list }
Union.t  = { fields : Field.t list }      (* same shape as Struct *)
Enum.t   = { fields : Enum_field.t list }

Field.t = { name : string; type_ : string; default : string option }
Enum_field.t = { name : string; value : int }
```

`Field.type_` is kept as a raw string (`"Optional[List[int]]"`) and parsed later by
`Type_expr_parse.of_string` when the code generator needs it. This avoids having to
design a jsont decoder for recursive type expressions.

## Custom Jsont Decoders for Ordered Fields

Standard jsont object decoding does not preserve declaration order. The schema needs
declaration order preserved (unions especially — the first field in source order becomes
the first constructor in the generated OCaml type).

Solution: decode as `Jsont.json` first, then match on `Jsont.Object (members, _)` and
map over the list. `members` is an association list preserving insertion order.

```ocaml
let list_jsont : t list Jsont.t =
  Jsont.map ~kind:"Fields"
    ~dec:(fun json ->
      match json with
      | Jsont.Object (members, _) ->
          List.map (fun ((name, _), value) ->
            match Jsont.Json.decode content_jsont value with
            | Ok (type_, default) -> make name type_ default
            | Error s -> Jsont.Error.msg Jsont.Meta.none s)
          members
      | _ -> Jsont.Error.msgf Jsont.Meta.none "Expected object for fields")
    Jsont.json
```

The same pattern is used for `Enum_field.list_jsont`.

## Schema Top-Level Decoding

The top-level YAML object is heterogeneous: most keys are type definitions, but
`SCHEMA_VERSION` and `TREESPEC_VERSION` are special sentinel keys. The decoder uses
`fold_left` over the raw member list and dispatches on the key name.

```ocaml
| "SCHEMA_VERSION" -> { acc with schema_version = ... }
| "TREESPEC_VERSION" -> { acc with treespec_version = ... }
| type_name -> { acc with types = String_map.add type_name td acc.types }
```

## Type Expression Parser

`Type_expr_lexer.mll` + `Type_expr_parser.mly` (ocamllex + Menhir) parse strings like
`"Optional[List[int]]"` into `Type_expr.t`:

```
Type_expr.t =
  | Str | Int | Float | Bool
  | List of t
  | Optional of t
  | Dict of t           (* Dict[str, T] — key is always str in PyTorch *)
  | Ref of string       (* reference to another named type *)
```

`Dict[K, V]` in the schema always has `K = str`, so the `Dict` node only stores `V`.

`Type_expr_parse.of_string` wraps the parser and raises `Failure` on parse error
(intentional: a parse error is a schema bug, not a runtime condition).

## YAML Quoting and String Defaults

YAML string quoting (`'False'`, `"False"`) is transparent to the parser — both
produce the OCaml string `"False"`. The code generator therefore must match both
`"False"` and `"'False'"` (the latter can appear if the YAML file was generated by a
tool that re-quotes the value). See `codegen_design.md` for the full pattern table.
