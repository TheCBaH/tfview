# Tensor C++ API → OCaml mapping plan

How `at::Tensor`'s *object* API (metadata, element access, dtype/device,
conversions) maps to OCaml. The ~2600 tensor *operations* (`add`, `conv2d`, …)
are out of scope — they are schema-generated by `bin/aten_ops_gen.ml` into
`atg_ops` (see `aten_op_schema.md`). This doc covers the hand-written half:
`atg_shim.{h,cpp}` / `function_description.ml` / `tensor.ml`.

## Constraints (why it is not 1:1)

| C++ feature | Boundary problem | Mapping rule |
|---|---|---|
| `Tensor`/`Scalar`/`IntArrayRef`/`Device`/`TypeMeta` non-POD by value | can't cross `extern "C"` | opaque `atc_tensor` handle; rest as C scalars / `(ptr,len)` |
| `item<T>()`, `data_ptr<T>()` templates | no symbol | monomorphize per dtype in the shim |
| overloads (`size()`/`size(d)`) | name clash | explicit C names |
| `ScalarType`/`Device`/`Layout`/`MemoryFormat` enums | — | int codes ↔ OCaml variants (`Scalar_type`) |
| `std::optional`, `IntArrayRef` | — | sentinel / `(has,val)`, `(ptr,len)` |
| **ATen throws `c10::Error`** | exceptions can't cross `extern "C"` (abort) | catch at the boundary → status + message → OCaml exn |
| autograd (`grad`, `requires_grad`) | inference-only | out of scope |
| `to`/`contiguous`/`clone`/`cpu` | **dispatch to kernels** | route via the op generator + add kernels to `build_archive.sh` |

Build-specific split: pure metadata accessors (`sizes`, `dtype`, `dim`,
`is_contiguous`) are inline/header-only — free to bind. Anything that *computes*
(`item` sync/copy, `to`/`contiguous`/`clone`) pulls into the static-dispatch
closure and follows the op cadence.

## Target OCaml surface

- `Dtype` GADT `('a,'b) t` tying the OCaml elt type to the Bigarray kind, with
  `to_int` (the c10 ScalarType code). Generalizes today's single `as_float32`.
- `Tensor` is the managed handle façade (RAII via `manage`, `live_count` —
  already in place):
  - lifecycle: `create ?dtype shape`, `of_bigarray`
  - metadata (free): `dim`, `shape`, `strides`, `numel`, `scalar_type`,
    `element_size`, `is_contiguous`, `defined`, `is_cpu`
  - data: `data : ('a,'b) Dtype.t -> t -> (...) Array1.t option`, `item_float`
  - debug: `pp`
  - `exception Error of string` raised at the `c10::Error` boundary

## Hand-written vs generated

- metadata/lifecycle: hand-write (small, irregular, stable).
- per-dtype accessors: tiny table over the ~8 supported dtypes (light codegen or
  a manual functor) — not worth the schema machinery.
- kernel-backed conversions (`to`/`contiguous`/`clone`): route through the op
  generator; add kernels on the op cadence.

## Error boundary primitive

Shim functions that can throw clear/catch and expose the message:

```cpp
#define ATC_TRY(sentinel, ...) \
  try { __VA_ARGS__ } \
  catch (const std::exception& e) { ::atc_detail::set_error(e.what()); return sentinel; }
const char* atc_last_error(void);   // thread-local; NULL when clean
```

OCaml `Tensor.check` raises `Tensor.Error (last_error ())` when a call returns
its sentinel (e.g. `atc_new` → null). NB: the *generated* `atg_` op wrappers are
**not** yet guarded — wrapping them in try/catch is a separate generator change.

## Phased roadmap

- **Phase 0 (done):** lifecycle, `new_`/`free`, `numel`/`dim`/`sizes`/`data_ptr`,
  float32 view, RAII (`manage`) + atomic live counter.
- **Phase 1 (in progress):** free metadata (`strides`, `scalar_type`,
  `element_size`, `is_contiguous`, `defined`, `is_cpu`) + the error boundary
  (`ATC_TRY` / `atc_last_error` / `Tensor.Error`) + `Tensor.create`. Header-only;
  no archive changes.
- **Phase 2:** `Dtype` GADT, generic `data`, `of_bigarray`, `item_float`/`item_int`.
- **Phase 3:** kernel-backed `contiguous`/`clone`/`to`/`cpu` via the op generator.
- **Phase 4:** `pp` via real ATen printing; `allclose`/`equal` for test asserts.

## Non-goals

Autograd, non-CPU devices, sparse/quantized/nested/named tensors, custom-stride
views beyond what kernels require.
